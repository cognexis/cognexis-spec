# 04. Attention Mechanisms

## 04.1 Purpose

Attention is the cross-token mixing mechanism used by the Prelude, Recurrent Core, and Coda. Cognexis is a decoder-only model, so all attention paths must be causal: a token may attend to itself and previous tokens, but never to future tokens. The default implementation uses grouped query attention (GQA) to reduce key/value memory while preserving multiple query heads.

## 04.2 Attention Variants

Cognexis supports these attention variants:

- `mha`: standard multi-head attention with independent query, key, and value heads.
- `gqa`: grouped query attention, the default for Prelude, Recurrent Core, and Coda.
- `mqa`: multi-query attention, an optional lower-memory variant with one key/value head group.
- `sliding_window`: optional local attention for long-context deployments.

The model config must specify the variant, number of query heads, number of key/value heads, head dimension, RoPE settings, and cache policy. If `num_kv_heads < num_attention_heads`, query heads are assigned to key/value groups.

## 04.3 Tensor Shapes

Given hidden input:

```text
X: [batch, seq_len, hidden_dim]
```

The projections produce:

```text
Q: [batch, num_q_heads, seq_len, head_dim]
K: [batch, num_kv_heads, seq_len, head_dim]
V: [batch, num_kv_heads, seq_len, head_dim]
```

The output is:

```text
O: [batch, seq_len, hidden_dim]
```

`hidden_dim` must equal `num_q_heads * head_dim` unless the implementation explicitly supports a separate projection dimension. For GQA, `num_q_heads` must be divisible by `num_kv_heads`.

## 04.4 Projection Parameters

The attention module owns four learned projections:

```text
W_q: [hidden_dim, num_q_heads * head_dim]
W_k: [hidden_dim, num_kv_heads * head_dim]
W_v: [hidden_dim, num_kv_heads * head_dim]
W_o: [num_q_heads * head_dim, hidden_dim]
```

Bias terms are optional and controlled by `attention_bias`. Bias-free projections are preferred for large decoder-only models unless compatibility with an existing checkpoint requires bias.

## 04.5 Computation

For each attention head:

```text
scores = (Q @ K^T) / sqrt(head_dim)
scores = scores + causal_mask + padding_mask + optional_attention_bias
weights = softmax(scores)
weights = dropout(weights)          # training only
context = weights @ V
output = concat(context_heads) @ W_o
```

Softmax must use a numerically stable implementation that subtracts the row maximum before exponentiation. Masked positions must receive an effectively negative infinite score before softmax. Implementations should use backend-native fused attention kernels when available, provided they preserve the same mask semantics.

## 04.6 Causal Masking

The causal mask for a sequence length `T` forbids attention from query position `i` to key position `j` when `j > i`. During decode with a KV cache, a single new query token attends to all cached keys plus itself. The implementation must account for absolute position offset so the new token does not accidentally mask valid cached positions.

When packed documents share one training block, an optional document-boundary mask may prevent tokens from attending across EOD boundaries. If enabled, this mask is combined with the causal mask.

## 04.7 Rotary Positional Encoding

RoPE is applied to `Q` and `K` after projection and before score computation. The rotation uses position IDs generated by the embedding/inference layer:

```text
Q_rot = rope(Q, position_ids)
K_rot = rope(K, position_ids)
```

The config must define:

- `rope_theta`
- `rope_scaling`, if long-context extrapolation is enabled
- `max_position_embeddings`
- whether RoPE is applied to all or part of each head dimension

RoPE must be applied consistently during training, prefill, and decode. Cached keys must store the already-rotated key vectors for their original positions, or the cache must store enough metadata to rotate them exactly once.

## 04.8 Grouped Query Attention

In GQA, multiple query heads share each key/value head. If there are `Hq` query heads and `Hkv` key/value heads, each key/value head serves:

```text
group_size = Hq / Hkv
```

query heads. This reduces KV cache memory from `Hq` heads to `Hkv` heads. The implementation may either expand K/V to query-head shape logically or use kernels that index key/value groups directly. Physical expansion should be avoided in memory-sensitive inference.

## 04.9 KV Cache Contract

The KV cache stores projected, position-encoded keys and values by layer, batch slot, token position, and key/value head:

```text
K_cache: [layers, batch, num_kv_heads, max_seq, head_dim]
V_cache: [layers, batch, num_kv_heads, max_seq, head_dim]
```

The cache must record:

- Current sequence length per batch slot.
- Maximum allocated sequence length.
- Layer identity.
- Dtype and device.
- Whether entries belong to Prelude, Recurrent Core, or Coda.

During recurrent loops, cache behavior depends on the selected recurrence design. If the recurrent core recomputes attention over updated hidden states, its per-loop K/V values are loop-dependent and must not be reused across loops as if they were static. If an optimized implementation reuses K/V, the recurrence equation must explicitly define that keys/values are anchored to a prior state. The default correctness-first behavior is to recompute recurrent-core Q/K/V for each loop.

## 04.10 Dropout and Determinism

Attention dropout applies to normalized attention weights during training only. It must be disabled for evaluation and inference. Distributed training must use reproducible RNG seeding so attention dropout is deterministic under checkpoint replay when requested.

Inference must be deterministic for greedy decoding at fixed precision and fixed loop count unless the backend is explicitly configured for nondeterministic kernels.

## 04.11 Long Context Options

For long contexts, a conforming implementation may use:

- Flash attention or memory-efficient exact attention.
- Sliding-window attention with global tokens.
- Paged KV cache allocation.
- RoPE scaling.
- KV cache quantization.

Any approximate or restricted attention variant must be declared in the config and evaluation reports. Exact full-context attention and sliding-window attention are not interchangeable for benchmark comparison.

## 04.12 Error Handling

Attention initialization must reject:

- `hidden_dim` not divisible by `num_attention_heads`.
- `num_attention_heads` not divisible by `num_kv_heads`.
- Unsupported head dimension for the selected kernel.
- Mask shapes that cannot broadcast to attention scores.
- Cache dtype/device mismatches.
- Position IDs outside supported RoPE range when extrapolation is disabled.

## 04.13 Rust Implementation Notes

Recommended module interface:

```rust
pub struct AttentionConfig {
    pub hidden_dim: usize,
    pub num_attention_heads: usize,
    pub num_kv_heads: usize,
    pub head_dim: usize,
    pub variant: AttentionVariant,
    pub rope: RopeConfig,
    pub dropout: f32,
}

pub trait Attention {
    fn forward(&self, x: &Tensor, ctx: AttentionContext<'_>) -> Result<Tensor>;
}
```

`AttentionContext` should carry masks, position IDs, cache handles, training/eval mode, and optional per-token active masks for token-wise scheduling. Avoid hidden global cache state; explicit context improves testability and serving isolation.

## 04.14 Validation

Attention tests must include:

- Shape tests for MHA, GQA, and MQA.
- Causal mask tests using small hand-computed examples.
- KV cache equivalence between full prefill and step-by-step decode.
- RoPE consistency for cached and non-cached paths.
- Dropout disabled in eval mode.
- Invalid config rejection.
- Numerical comparison against a simple reference implementation for small tensors.
