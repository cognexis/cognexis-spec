# 26. Implementation Outline and Build

## 26.1 Purpose

This document describes a practical Rust project structure for implementing Cognexis. It defines module boundaries, major traits, build expectations, testing strategy, and deployment artifacts. The outline is intentionally modular so research implementations can start small and production implementations can replace backends without changing high-level contracts.

## 26.2 Repository Layout

Recommended layout:

```text
Cargo.toml
build.rs
configs/
  train.yaml
  eval.yaml
  serve.yaml
src/
  lib.rs
  bin/
    cognexis_train.rs
    cognexis_eval.rs
    cognexis_serve.rs
  config/
  tokenizer/
  tensor/
  model/
  scheduler/
  training/
  inference/
  evaluation/
  safety/
  telemetry/
tests/
  integration/
benches/
docs/
```

`src/tensor` should isolate the tensor backend. The rest of the codebase should depend on traits and typed tensors rather than directly calling backend-specific APIs everywhere.

## 26.3 Model Modules

Recommended model layout:

```text
src/model/
  mod.rs
  embedding.rs
  rope.rs
  norm.rs
  attention.rs
  ffn.rs
  block.rs
  prelude.rs
  recurrent.rs
  coda.rs
  lm_head.rs
  checkpoint.rs
```

The top-level model owns:

```rust
pub struct CognexisModel {
    pub embeddings: Embedding,
    pub prelude: Prelude,
    pub recurrent: RecurrentCore,
    pub coda: Coda,
    pub final_norm: Norm,
    pub lm_head: LmHead,
}
```

Forward methods should accept explicit contexts for masks, caches, mode, and scheduler options.

## 26.4 Tensor Backend Boundary

The tensor backend must provide:

- Tensor allocation and shape metadata.
- Matrix multiplication.
- Gather/embedding lookup.
- Elementwise operations.
- Reductions.
- Softmax and attention kernels.
- Autograd for training or a bridge to an autograd framework.
- Device transfer.
- Serialization support.

Backend-specific optimized kernels should live behind interfaces. This allows a CPU reference backend for tests and a GPU backend for performance.

## 26.5 Configuration and Errors

Use typed configuration structs with validation. Use structured errors:

```rust
#[derive(thiserror::Error, Debug)]
pub enum CognexisError {
    #[error("invalid config: {0}")]
    InvalidConfig(String),
    #[error("shape mismatch: expected {expected}, got {actual}")]
    ShapeMismatch { expected: String, actual: String },
    #[error("backend error: {0}")]
    Backend(String),
}
```

Avoid panics for user/config/runtime errors. Panics are acceptable only for internal invariants that indicate programmer mistakes and should be rare.

## 26.6 Tokenization Pipeline

Tokenization module responsibilities:

- Load tokenizer artifacts and manifest.
- Encode/decode text.
- Render chat templates.
- Validate special tokens.
- Provide streaming decode helpers.

The inference API should accept either raw text or pre-tokenized IDs, but raw text must flow through the same tokenizer manifest bundled with the checkpoint.

## 26.7 Training Pipeline

Training components:

```text
training/
  trainer.rs
  data/
  curriculum.rs
  optimizer.rs
  loss.rs
  checkpoint.rs
  distributed.rs
```

Training loop:

1. Load resolved config.
2. Initialize tokenizer, model, optimizer, data loader, curriculum.
3. Sample batch and loop metadata.
4. Run forward pass.
5. Compute LM loss and auxiliary losses.
6. Backpropagate with gradient accumulation/checkpointing.
7. Clip gradients.
8. Optimizer step.
9. Log metrics.
10. Save checkpoints and run validation.

The loop curriculum state must be checkpointed with the optimizer.

## 26.8 Inference Pipeline

Inference components:

```text
inference/
  generate.rs
  prefill.rs
  decode.rs
  cache.rs
  sampler.rs
  batching.rs
  streaming.rs
```

Generation flow:

1. Validate request and safety policy.
2. Tokenize prompt.
3. Allocate cache.
4. Prefill.
5. Repeatedly decode, schedule loops, sample, filter, and stream.
6. Release cache.
7. Return final text and diagnostics.

The API should expose both blocking and streaming generation.

## 26.9 Scheduler Modules

```text
scheduler/
  mod.rs
  fixed.rs
  rule_based.rs
  value_head.rs
  token_wise.rs
  budget.rs
  diagnostics.rs
```

Schedulers implement a shared trait and can be selected by config. The recurrent core should receive a scheduler trait object or generic parameter, not a concrete scheduler type hardcoded into the model.

## 26.10 Evaluation Modules

```text
evaluation/
  harness.rs
  tasks/
  metrics.rs
  loop_scaling.rs
  ablation.rs
  report.rs
```

Evaluation should produce machine-readable JSONL plus optional plots generated by external tools. Keep metric formulas in code with unit tests.

## 26.11 Safety and Telemetry Modules

```text
safety/
  input_filter.rs
  output_filter.rs
  code_sandbox.rs
  policy.rs
telemetry/
  logging.rs
  metrics.rs
  tracing.rs
```

Safety wraps inference rather than being embedded inside tensor kernels. Telemetry should use structured fields and avoid raw content by default.

## 26.12 Checkpoint Format

A checkpoint bundle should include:

```text
checkpoint/
  model.safetensors
  optimizer/              # training checkpoints only
  config.resolved.yaml
  tokenizer.model
  tokenizer.json
  scheduler.json
  metadata.json
```

`metadata.json` should include:

- Checkpoint step.
- Git commit or build ID.
- Schema version.
- Tensor dtype.
- Parameter count.
- Training tokens seen.
- Data manifest checksum.
- Hardware/backend summary if available.

Use a tensor format that preserves names, dtypes, shapes, and supports sharded checkpoints for large models.

## 26.13 CLI Commands

Recommended commands:

```text
cognexis train --config configs/train.yaml
cognexis eval --config configs/eval.yaml --checkpoint checkpoint/
cognexis eval loop-scaling --config configs/eval.yaml --depths 1,2,4,8
cognexis serve --config configs/serve.yaml --checkpoint checkpoint/
cognexis tokenize --tokenizer tokenizer.json --text "..."
cognexis inspect-checkpoint checkpoint/
```

CLI commands should print clear validation errors and write resolved configs to output directories.

## 26.14 Build and Features

Cargo features may include:

```text
cpu
cuda
wgpu
distributed
quantization
safetensors
server
```

Default features should build a CPU reference implementation if feasible. GPU features may require external toolchains and should be checked by `build.rs` or a doctor command.

## 26.15 Testing Strategy

Test layers:

- Unit tests for tokenizer, masks, norms, attention, FFN, scheduler, metrics.
- Golden tests for tokenizer and config parsing.
- CPU reference tests for model forward.
- Integration tests for prefill/decode equivalence.
- Training smoke test on a tiny model.
- Evaluation smoke test on tiny fixtures.
- Serialization round-trip tests.
- Performance benchmarks for attention, FFN, recurrence, and cache.

Every optimized backend path should have a small reference comparison.

## 26.16 Deployment

Deployment artifacts:

- Model checkpoint bundle.
- Tokenizer artifact.
- Resolved serving config.
- Container image or binary.
- Hardware requirements.
- Safety policy config.
- Monitoring dashboard definitions.

Serving should support graceful shutdown, request cancellation, health checks, and cache cleanup.

## 26.17 Validation

The implementation is build-ready when:

- `cargo test` passes for CPU/reference features.
- Config validation rejects known bad examples.
- A tiny model can train for several steps.
- Prefill/decode equivalence passes.
- Fixed and adaptive loop modes run end to end.
- Evaluation emits valid result JSONL.
- Serving enforces token and loop budgets.
- Checkpoint save/load round trip preserves outputs for a fixed seed.
